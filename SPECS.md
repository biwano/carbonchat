# Specifications: Carbonchat

## Overview
Carbonchat is an **AI-powered knowledge chatbot** with realtime capabilities. It maintains a dynamic **knowledge base** built by scraping and synthesizing information from the internet using AI.

**Core Concept**: 
Users maintain two related Supabase tables:
- `document_types`: Reusable templates containing transformation instructions (how the AI should process researched information). Each type has an `ai` boolean flag that determines whether its documents are AI-researched or authored manually by the user.
- `subjects`: Defines the core topic of a document. Contains a name and a detailed content description of what the document should be about.
- `documents`: Individual knowledge entries that reference a document type and a subject (for AI types) or hold user-authored `content` (for non-AI types).

For **AI document types** (`ai = true`), an API endpoint triggers an AI agent (via OpenRouter) to perform web research and generate knowledge according to the selected document type's rules and the associated subject's content. For **non-AI document types** (`ai = false`), documents are purely manual: the user writes the `content` directly and no subject is required.

When a user interacts with the chatbot, **all documents are retrieved** and injected — along with their corresponding document type instructions — as a rich system prompt. The provided information acts as the **primary source of truth**, though the assistant can use additional information from the internet to elaborate on answers as long as it doesn't contradict the primary context.

## Core Features
- **AI Chatbot**: Primary interface; loads **all documents** + their associated document type instructions as rich system context/preprompt for every response. **The chatbot maintains full conversation history**, sending the entire message chain to the AI for every new response.
- **Administration Panel** (renamed from "Settings"): Section for knowledge management at `/admin`:
  - **Document Types Management**: Full CRUD support (create, read, **update**, **delete**) for transformation instruction templates. Each type exposes an **`ai` toggle** that controls whether documents of this type are AI-researched (default) or manually authored. `transformation_instructions`, **`additional_sources`**, and **`source_relevance_factors`** are only shown and edited for AI types; for manual types they are hidden in the form and on the type card, and stored as an empty string. All operations are performed **directly from the frontend** using **TanStack Query** for efficient data fetching and cache management.
- **Subjects Management**: CRUD support for subjects. Each subject defines a **name** and **content** (the topic/source info). Managed using **TanStack Query**.
- **Documents Management**: CRUD with behavior that depends on the selected document type's `ai` flag. Managed using **TanStack Query** and **Supabase Cache Helpers**.
    - For **AI document types** (`ai = true`): clicking the **"Start Research"** button in the create dialog triggers a two-step process: first, the document record is created in Supabase with a placeholder state; second, the `POST /api/documents/scrape` endpoint is called to perform AI-powered research. Content and **sources** are generated by the AI agent based on the subject's content, the document type's **`additional_sources`**, and rated according to **`source_relevance_factors`**. The edit dialog allows manual updating of **name**, **document type**, **subject**, **content**, and **sources**. Manual edits are saved directly; clicking **"Refresh Content"** triggers the scraper again, overwriting content and sources with fresh research. The Documents panel also features a **"Refresh All Content"** button that concurrently triggers the scraper for all AI-researched documents.
    - For **non-AI document types** (`ai = false`): documents are purely manual. The create button is labeled **"Create Document"**. The dialog asks for **name**, **document type**, **content**, and **sources** (both editable at creation). The `subject` field is hidden. No "Refresh Content" action is offered.
  - **Help**: Static documentation tab that explains the mental model of `document_types` vs `documents`, recommended workflow (define a type → create a document → research → chat), how content flows into the chat as system prompt, and best-practice tips for writing transformation instructions and **source relevance factors**.
- **Document Scraper**: `POST /api/documents/scrape` endpoint using OpenRouter to research a **subject** and synthesize per `document_type` rules. Returns a structured JSON response containing synthesized **content** and the **sources** used. The scraper uses the instructions in **`source_relevance_factors`** to rate the pertinence of each source (alongside age) and adds a note reflecting its relevance to each returned source. Supports both initial creation and updating (refreshing) existing documents. Only applicable to documents whose type has `ai = true`; the endpoint rejects requests targeting documents of non-AI types.
- **Real-time Chat Interface**: Modern streaming responses where the curated information serves as the primary source of truth. **The chat takes into account the whole conversation history** to provide contextual and coherent responses. When a user submits a message, the AI response is **streamed token-by-token from the server to the client over HTTP streaming** and rendered progressively in the chat bubble as it is generated (no waiting for the full response before the first character appears). The UI must clearly indicate an in-progress / streaming state. While a response is streaming, the user can **cancel/stop** generation; any tokens already received are kept as the (partial) assistant message. If the stream errors mid-way, the partial content so far is preserved and an inline **error message in red** is appended to that same assistant bubble. The assistant is encouraged to use its general knowledge to elaborate on the provided facts.
- **Unrestricted Public Access**: Row Level Security (RLS) is configured to allow **unauthenticated (anon) CRUD access** to both `document_types` and `documents` tables (useful for development/testing).

## Technical Stack

- **Frontend**: Next.js 15 (App Router), TypeScript, Tailwind CSS, shadcn/ui, **TanStack Query (React Query)**
- **Backend**: Next.js API Routes + Server Actions
- **Database**: Supabase (PostgreSQL) — **used exclusively for the knowledge base**
- **Auth**: Supabase Auth
- **Real-time**: Supabase Realtime (for chat presence and live updates)
- **AI Integration**: **OpenRouter** via the official `openai` SDK (configured with `baseURL: "https://openrouter.ai/api/v1"` and required headers)
- **Web Scraping / Research**: AI agents via OpenRouter (with search capabilities). The agent explicitly checks **`additional_sources`** defined in the `document_type` to supplement its research.
- **Deployment**: Vercel (with Edge Functions where possible)

## Architecture

**Core Principle**: The database (Supabase) is **used only to store the knowledge base**. Chat history is kept in the client's state and passed to the API on each request.

### Key Components
1. **Document Scraper Service**: `POST /api/documents/scrape` endpoint that uses OpenRouter (with search capabilities) to research a topic based on a `subject`'s content and any **`additional_sources`** defined in the document type (e.g., specific blogs or social media accounts), then synthesize knowledge according to a `document_type`'s `transformation_instructions`.
2. **Knowledge Base Layer**: Three Supabase tables (`document_types`, `subjects`, and `documents`) with proper foreign key relationships. **All data fetching and mutations are handled via TanStack Query and Supabase Cache Helpers** to ensure robust state management, caching, and loading states. All schema changes must use timestamp-prefixed migration files in `supabase/migrations/` (e.g. `20260415114130_initial_schema.sql`).
3. **Chat Engine**: On every user message, the system:
   - Loads all documents (with joined type instructions)
   - Builds a rich system prompt containing all document content + transformation rules
   - **Sends the full conversation history (messages) along with the system prompt to OpenRouter.**
   - Streams the OpenRouter completion back to the client in real time, so the client can render the response incrementally as tokens arrive.
4. **UI Structure**: 
   - Full-page **Chat** interface at `/` (no tab selector).
   - Dedicated full-page **Administration** route at `/admin` (replaces previous Settings/Administration tab selector and separate panels; "Back to Chat" button removed from admin header for cleaner full-page experience).
   - **Documents Management**: Feature-rich document list with a **"Refresh All Content"** button in the header. Each document is rendered using a dedicated **`DocumentCard`** component that handles its own states (loading, refreshing, editing).
   - **Theme**: Default shadcn/ui dark theme only. Remove all custom/hardcoded colors (e.g. `violet-*`, `zinc-*`, `emerald-*`, specific bg-*/text-*/border-* Tailwind color classes). Use only semantic theme classes (`bg-background`, `text-foreground`, `bg-card`, `border-border`, `bg-primary`, `text-primary-foreground`, `bg-muted`, etc.). Apply `dark` class by default to `<html>` in layout. Simplify `globals.css` to rely on Tailwind v4 + shadcn defaults without extensive custom oklch color palette.
5. **AI Service Layer**: Centralized OpenAI client configured for OpenRouter.

### Supabase Data Model

**Table 1: `document_types`** (stores reusable transformation rules)
```sql
- id (uuid, primary key)
- name (text)                    -- e.g. "Technical Documentation", "Market Research", "Wikipedia-style Summary"
- transformation_instructions (text)  -- The prompt that tells the AI how to transform raw data (AI types only; empty string for manual types)
- additional_sources (text)         -- Optional sources (blogs, social media accounts, etc.) for the AI agent to research
- source_relevance_factors (text)   -- Instructions for the AI to rate the pertinence of a source (e.g. "Prioritize official documentation", "Ignore forums")
- description (text)
- ai (boolean, default true)     -- If true, documents of this type are AI-researched; if false, documents are manually authored
- created_at (timestamptz)
```

**Table 2: `subjects`** (stores document topics)
```sql
- id (uuid, primary key)
- name (text)                    -- e.g. "React Hooks", "Vercel AI SDK"
- content (text)                 -- Detailed description of what the subject is about
- created_at (timestamptz)
```

**Table 3: `documents`** (references document type and subject)
```sql
- id (uuid, primary key)
- name (text)                    -- Short identifier for this document/knowledge piece
- document_type_id (uuid, fk → document_types)
- subject_id (uuid, fk → subjects, nullable) -- Nullable for manual types
- content (text)                 -- The synthesized knowledge (AI-generated for AI types, user-authored for non-AI types)
- sources (text)                 -- Resources and URLs used for research (AI-generated for AI types, user-authored for non-AI types)
- created_at (timestamptz)
- updated_at (timestamptz)
- metadata (jsonb)
```

**Relationships**: 
- One `document_type` can be used by many `documents`.
- One `subject` can be used by many `documents`.
**Security**: RLS policies — primarily admin/owner access for knowledge management. Public read access for the chatbot if desired.

## Non-Functional Requirements
- Fast knowledge retrieval and prompt construction
- High-quality AI-generated knowledge (good research + synthesis)
- Low latency chat responses with streaming
- Privacy-focused (especially around web scraping and user queries)
- Responsive, modern UI using **default shadcn/ui dark theme only** (mobile-first, semantic color classes, no custom color palette)
- Accessibility compliant
